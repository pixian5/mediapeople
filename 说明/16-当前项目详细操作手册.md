# 缘定传媒人当前项目详细操作手册

本文档记录当前项目的真实操作方式，按“本地开发代码、提交 Git、服务器自动拉取、线上验证”的流程编写。项目目前是开发测试版，允许重构和清理旧演示数据，但每次修改后必须验证客户小程序端、红娘工作台、管理后台三端主流程。

## 1. 当前项目一句话说明

`mediapeople` 是一个婚恋/牵线原型系统，包含：

- 客户小程序端：会员注册登录、资料维护、浏览异性、付费成为红娘 VIP、申请牵线、与红娘聊天、经红娘批准后与对方会员聊天。
- 红娘工作台：红娘登录、查看牵线请求、联系男方、联系女方、开通/关闭双方会员互聊、与会员一对一聊天。
- 管理后台：管理员登录、查看客户/红娘/机构/兑换码/分成/成交数据。
- API 服务：Node.js Express，连接 PostgreSQL。
- 数据库：PostgreSQL Docker 容器。
- 前端静态页面：HTML/CSS/JS，由 Nginx Docker 容器提供。
- uniapp 小程序代码：在 `uniapp/` 目录，用于 H5/微信小程序构建。

## 5. 修改前必须先看状态

每次开始前先看 Git 状态：

```bash
git status --short
git log --oneline -5
```

如果看到不是自己改的文件，不要随便还原。先判断是否和当前任务有关。除非明确确认，否则不要执行：

```bash
git reset --hard
git checkout -- 文件名
```

## 6. 本地检查命令

修改 `app.js` 后：

```bash
node --check app.js
```

修改 `server/index.js` 后：

```bash
node --check server/index.js
```

修改部署脚本后：

```bash
bash -n deploy/auto-deploy.sh
bash -n deploy/remote-deploy.sh
```

检查空白和补丁错误：

```bash
git diff --check
```

生成静态 HTML 版本号：

```bash
npm run render:static
```

这会读取 `index.html`、`mini.html`、`matchmaker.html`、`admin.html`，把里面的 `__ASSET_VERSION__` 替换成当前 Git 短提交号，并输出到 `dist/`。

`dist/` 被 `.gitignore` 忽略，不提交。服务器部署时会重新生成。

uniapp H5 构建：

```bash
cd /Users/x/code/mediapeople/uniapp
npm run build:h5
```

uniapp 微信小程序构建：

```bash
cd /Users/x/code/mediapeople/uniapp
npm run build:mp-weixin
```

如果看到 Sass deprecated warnings，通常是依赖警告，不一定影响构建结果。真正需要处理的是构建失败、语法错误、页面运行错误。

## 8. 当前服务器自动部署流程

服务器自动部署脚本：

```text
/opt/mediapeople/deploy/auto-deploy.sh
```

脚本当前做的事情：

```text
1. cd /opt/mediapeople
2. git pull origin master
3. 运行 scripts/render-static.mjs，生成 dist/*.html 和 dist/version.json
4. 如果 uniapp/ 有变化，构建 uniapp H5
5. 如果 server/ 有变化，重建并重启 API 容器
6. 重启各个前端 Nginx 容器
7. 输出部署日志
```

手动在服务器触发自动部署：

```bash
ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech 'cd /opt/mediapeople && bash deploy/auto-deploy.sh'
```

部署后确认线上静态版本号：

```bash
ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech \
  'cd /opt/mediapeople && grep -R "app.js?v=" -n dist | head -10'
```

正常结果类似：

```text
dist/matchmaker.html:128:    <script src="/app.js?v=47f2d41"></script>
dist/mini.html:473:        <script src="/app.js?v=47f2d41"></script>
dist/admin.html:176:      <script src="/app.js?v=47f2d41"></script>
dist/index.html:766:      <script src="/app.js?v=47f2d41"></script>
```

这里的 `47f2d41` 应该等于服务器当前 Git 短提交号：

```bash
ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech \
  'cd /opt/mediapeople && git rev-parse --short HEAD'
```

## 9. 为什么以前会出现旧脚本缓存

之前页面加载的是：

```text
app.js?v=7392f4f
```

但服务器代码已经是更新的提交。这说明问题不是浏览器单纯缓存，而是服务器挂载的 `dist/*.html` 没有重新生成。

当前修复方式：

```text
HTML 模板里写：/app.js?v=__ASSET_VERSION__
部署时运行：scripts/render-static.mjs
生成 dist/*.html：/app.js?v=当前 Git 短提交号
Nginx 挂载 dist/*.html
浏览器看到新 query，自然重新拉新 app.js
```

排查缓存问题按这个顺序：

```bash
# 1. 看线上页面当前脚本版本
curl -sk https://uk.sbbz.tech:9446/ | grep 'app.js?v='

# 2. 看服务器 dist 当前版本
ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech \
  'cd /opt/mediapeople && grep -R "app.js?v=" -n dist | head'

# 3. 看服务器 Git 版本
ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech \
  'cd /opt/mediapeople && git rev-parse --short HEAD'

# 4. 重新生成并部署
ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech \
  'cd /opt/mediapeople && bash deploy/auto-deploy.sh'
```

如果第 1 步和第 2 步都还是旧版本，说明服务器没有生成新 `dist`。

如果第 2 步是新版本但浏览器还是旧版本，尝试强制刷新页面，或确认访问的是不是正确端口。

## 11. 客户端完整操作流程

线上入口：

```text
https://uk.sbbz.tech:9446/
```

### 11.1 登录已有会员

1. 打开客户小程序入口。
2. 默认可能进入“我的”页。
3. 找到“已有客户账号”。
4. 在下拉框选择一个会员，例如 `林安`。
5. 点击“登录”。
6. 登录成功后，顶部应显示会员名和 VIP 状态。

正常现象：

```text
顶部显示：林安
如果已绑定红娘 VIP，显示：1 位红娘 VIP
如果未开通，显示：普通用户
```

### 11.2 浏览异性会员

1. 点击底部或顶部 Tab 的“筛选”。
2. 系统会按当前用户性别默认展示异性。
3. 可选择：

```text
性别：看女生 / 看男生
城市：全部 / 上海 / 广州 / 杭州 / 南京 / 苏州 / 成都 / 深圳
年龄：不限 / 22-28 / 29-35 / 36-45
```

4. 卡片里应显示姓名、性别、年龄、城市、职业、自我介绍。
5. 点击“换一位”可切换展示对象。

### 11.3 申请牵线

在异性资料卡上：

1. 确认卡片中显示“选择联系的红娘”。
2. 如果对方绑定多个红娘，选择想要申请的红娘。
3. 点击“申请牵线”。

当前规则：

```text
会员看到异性后，可以付费成为该异性绑定红娘的专属 VIP。
如果异性绑定多个红娘，需要选择一个红娘。
成为该红娘 VIP 后，可以和该红娘绑定的异性会员发起牵线。
申请创建后，会员能和红娘聊天。
会员之间必须等红娘批准后才出现聊天入口。
```

正常现象：

```text
普通用户申请时，会自动模拟付费成为该红娘 VIP。
申请成功后出现提示：已通知红娘为你和某某牵线。
“消息”页出现一条牵线请求。
```

### 11.4 查看牵线请求

1. 点击“消息”。
2. 查看“我的牵线请求”。
3. 请求会显示：

```text
状态：待红娘联系 / 联系男方 / 联系女方 / 来和双方对话
双方姓名
负责红娘
对方微信，只有红娘联系完成后才显示
会员互聊状态：等待红娘开通 / 已开启
```

### 11.5 会员和红娘聊天

申请牵线成功后：

1. 在“消息”页点击“联系红娘”。
2. 页面下方出现聊天窗口。
3. 输入消息。
4. 点击“发送”。

正常现象：

```text
聊天窗口显示“我”发送的消息。
红娘工作台对应会员聊天线程里能看到该消息。
红娘回复后，会员端聊天窗口能看到红娘名字和回复内容。
```

### 11.6 会员和会员互聊

会员互聊前提：

```text
红娘已开通双方沟通。
```

操作：

1. 会员进入“消息”页。
2. 如果红娘已开通，会看到“与对方互聊”。
3. 点击“与对方互聊”。
4. 输入消息并发送。

正常现象：

```text
聊天标题显示对方会员姓名，例如“孟晚棠”。
聊天说明显示“互聊已开启 · 林安 与 孟晚棠”。
消息发送后，会员互聊列表预览更新为最新消息。
对方会员账号登录后，也能看到这条会员互聊线程和消息预览。
```

### 11.7 我的资料

当前“资料”已经合并到“我的”选项卡。

操作：

1. 进入“我的”。
2. 找到“我的资料”。
3. 修改姓名、性别、年龄、城市、职业、微信、自我介绍、择偶要求。
4. 选择要委托的红娘。
5. 如需同步到所有已绑定红娘，勾选“一键同步到所有已绑定红娘”。
6. 点击“保存资料并提交审核”。

规则：

```text
会员可以绑定多个红娘，成为多个红娘的 VIP。
会员资料和每个红娘绑定。
会员可以在红娘 1 发布资料 1，在红娘 2 发布资料 2，互不冲突。
勾选一键同步后，会把当前资料同步给所有已绑定红娘。
会员修改资料后，需要绑定红娘审核后发布。
```

ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech \
