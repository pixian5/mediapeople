## 模态弹窗通用行为

### 弹窗结构

所有模态弹窗使用相同的 HTML 结构：

```html
<div class="modal-backdrop" id="xxxModal" style="display: none;">
  <div class="modal-content">
    <div class="modal-header">
      <h2>标题</h2>
      <button class="modal-close" id="closeXxxModalBtn">&times;</button>
    </div>
    <div id="xxxModalBody">
      <!-- 内容 -->
    </div>
  </div>
</div>
```

### 弹窗显示/隐藏

**显示弹窗**：

```javascript
const modal = $("#xxxModal");
modal.style.display = "flex";
modal.classList.add("show");
```

**隐藏弹窗**：

```javascript
const modal = $("#xxxModal");
modal.style.display = "none";
modal.classList.remove("show");
```

### 关闭弹窗的三种方式

1. **点击 × 按钮**：触发 close 按钮的 click 事件
2. **点击弹窗背景**：检测 `event.target === event.currentTarget`，即点击的是 .modal-backdrop 本身
3. **代码调用**：直接设置 display 和 class

## 实名认证模态框

### 触发条件

在"我的"Tab 点击"实名认证"菜单项（`#realNameVerificationMenuItem`）。

### 未认证状态

**表单字段**：

| 字段 | 类型 | 必填 | 校验规则 |
|------|------|------|----------|
| 真实姓名 | input | 是 | 非空 |
| 身份证号 | input | 是 | 18位身份证格式：`/^[1-9]\d{5}(18|19|20)\d{2}((0[1-9])\|(1[0-2]))(([0-3][0-9]))\d{3}[0-9Xx]$/` |
| 补填手机号 | input[tel] | 条件 | 仅注册时未填手机号时显示，11位数字 |

**补填手机号逻辑**：

```
1. 检查当前用户是否有 phone 字段
2. 如无 phone → 显示"补填手机号"输入框，设为必填
3. 如有 phone → 隐藏该输入框，设为非必填
```

**提交流程**：

```
1. 点击"提交实名认证"
2. 校验姓名非空→为空显示"请填写真实姓名和身份证号"
3. 校验身份证号非空→同上
4. 校验身份证号格式→不合法显示"请输入合法的18位身份证号"
5. 如需补填手机号→校验手机号格式→不合法显示"请输入合法的11位手机号"
6. API 可用→调用 POST /api/client/real-name { realName, idCard, phone }
7. 成功→显示"实名认证成功！"，1.2秒后自动关闭弹窗
8. 失败→显示错误提示
```

### 已认证状态

**显示内容**：

| 元素 | 显示内容 | 脱敏规则 |
|------|----------|----------|
| ✅ 图标 | 大号绿色勾 | - |
| 标题 | "已通过实名认证" | - |
| 描述 | "您的身份信息已由国家公安接口安全加密验证" | - |
| 真实姓名 | 林*安 | 姓名中间加 *，首尾保留 |
| 身份证号 | 3101***********1234 | 前4后4，中间用 * 填充 |
| 认证手机 | 138****0001 | 前3后4，中间用 * 填充 |

**脱敏函数**：

```javascript
maskName("林安")     → "林*安"
maskName("张三丰")   → "张*丰"
maskIdCard("310101199001011234") → "3101***********1234"
maskPhone("13800000001") → "138****0001"
```

### 关闭弹窗

| 触发方式 | 行为 |
|----------|------|
| 点击 × 按钮 | 关闭弹窗 |
| 点击弹窗背景 | 关闭弹窗 |
| 提交成功后 1.2 秒 | 自动关闭弹窗 |

## 红娘聊天模态框

### 触发条件

1. 在红娘工作台点击聊天线程卡片
2. 在红娘工作台点击"来和双方对话"按钮

### 弹窗内容

**头部**：

| 元素 | 类型 | 显示内容 |
|------|------|----------|
| 标题 | h2 | 会话名称（如"林安 (一对一沟通)"） |
| 描述 | span | 会话描述 |
| × 关闭按钮 | 按钮 | 关闭弹窗 |

**消息列表**：

| 元素 | 显示内容 |
|------|----------|
| 空消息提示 | "还没有聊天消息。" |
| 消息列表 | 每条消息显示：发送者名称、内容、时间 |

**消息样式**：

| 消息类型 | 对齐 | 背景 | 圆角 |
|----------|------|------|------|
| 自己发送的消息 | 右对齐 | 蓝色渐变 | 16px 16px 4px 16px |
| 对方发送的消息 | 左对齐 | 白色 | 16px 16px 16px 4px |

**底部输入区**：

| 元素 | 类型 | 操作后行为 |
|------|------|------------|
| 消息输入框 | input | placeholder "给会员发送消息" |
| 发送 | 按钮 | 调用 `sendMatchmakerChatMessage()` |

### 发送消息流程

```
1. 获取当前活跃线程（activeMatchmakerChatThreadId）
2. 获取输入框内容
3. 内容为空→不处理
4. API 可用→调用 POST /api/chat/threads/:id/messages { content }
5. 成功→清空输入框，更新 state，重新渲染
6. 失败→显示错误提示
7. API 不可用→本地追加消息，保存 state
```

### 关闭弹窗

```
1. 点击 × 按钮 或 点击弹窗背景
2. 设置 matchmakerChatModalOpen = false
3. 设置 activeMatchmakerChatThreadId = null
4. 重新渲染（弹窗隐藏）
```

### 页面卸载同步

**触发条件**：`pagehide` 事件

**行为**：

```
1. 检查 API 是否可用
2. 如可用→调用 syncRemoteState({ keepalive: true, notify: false })
3. keepalive: true 确保请求在页面关闭前完成
4. notify: false 不显示同步失败提示
```

### 4 秒轮询同步

**触发条件**：页面加载后启动，每 4 秒执行一次

**行为**：

```
1. 检查 API 是否可用
2. 如可用→调用 loadRemoteState()
3. 成功→比较 JSON.stringify(remote) !== JSON.stringify(local)
4. 如不同→更新本地 state + localStorage + renderAll()
5. 如相同→无操作
6. 失败→跳过本次，不影响下次轮询
```

**轮询目的**：

- 实现多端实时同步
- 客户端操作后，红娘端/管理后台自动更新
- 红娘端操作后，客户端自动更新

## 事件委托机制

### safeBind 函数

```javascript
function safeBind(selector, event, handler) {
  const el = $(selector);
  if (el) el.addEventListener(event, handler);
}
```

- 安全绑定事件，元素不存在时不报错
- 用于绑定页面上所有静态元素的事件

### 事件委托

对于动态生成的元素（如聊天消息、请求卡片），使用事件委托：

```javascript
// 在父容器上监听 click 事件
safeBind("#notificationList", "click", (event) => {
  // 使用 closest() 查找最近的匹配元素
  const button = event.target.closest("[data-contact-request]");
  if (button) {
    // 获取自定义属性
    const requestId = button.dataset.contactRequest;
    const side = button.dataset.contactSide;
    contactRequestSide(requestId, side);
    return;
  }
});
```

## 弹窗层级与遮罩管理

### z-index 层级规范

```
层级从低到高：

1. 页面内容        z-index: 1
2. 顶部导航栏      z-index: 10
3. 侧边栏          z-index: 20
4. 下拉菜单        z-index: 100
5. Toast 提示      z-index: 9999
6. 模态弹窗遮罩    z-index: 10000
7. 模态弹窗内容    z-index: 10001
8. 弹窗中的弹窗    z-index: 10010/10011
9. 图片预览弹窗    z-index: 99999
```

**层级说明**：

| 层级 | 元素类型 | z-index 值 | 说明 |
|------|---------|-----------|------|
| 基础层 | 页面正文 | 1 | 默认层级 |
| 导航层 | 顶部/侧边导航 | 10-20 | 页面内固定导航 |
| 浮层 | 下拉菜单、Tooltip | 100 | 临时浮动元素 |
| 提示层 | Toast、Loading | 9999 | 全局提示，不阻塞操作 |
| 弹窗层 | 模态弹窗 | 10000-10100 | 阻塞式弹窗 |
| 最高层 | 图片预览、确认弹窗 | 99999 | 最顶层显示 |

### 多弹窗堆叠处理

**堆叠规则**：

```
1. 同一时间只允许一个主弹窗（业务弹窗）
```

### 背景滚动锁定

**锁定机制**：

```
弹窗打开时：
1. 记录当前滚动位置
2. 给 body 添加 class: .modal-open
3. 设置 body { overflow: hidden; position: fixed; width: 100%; }
4. 设置 body 的 top 值为负的滚动位置

弹窗关闭时：
1. 移除 body 的 .modal-open class
2. 恢复 body 的 overflow 和 position
3. 滚动到记录的位置
```

**CSS 实现**：

```css
body.modal-open {
  overflow: hidden;
  position: fixed;
  width: 100%;
  height: 100%;
}
```

**特殊情况处理**：

| 场景 | 处理方式 |
|------|---------|
| 多个弹窗堆叠 | 只有第一个弹窗打开时锁定，最后一个关闭时解锁 |
| 弹窗内有滚动 | 弹窗内容区域独立滚动，不影响背景 |
| 移动端 | 额外处理 touchmove 事件防止穿透 |

## 全局状态同步

### localStorage 变更事件

**存储事件（storage event）**：

```
触发时机：
- 当其他标签页修改 localStorage 时
- 当前标签页不会触发自己修改的 storage 事件

监听方式：
window.addEventListener('storage', (e) => {
  if (e.key === 'mediapeople_state') {
    // 其他标签页更新了数据
    syncStateFromStorage();
  }
});
```

**事件对象属性**：

| 属性 | 说明 |
|------|------|
| `e.key` | 被修改的键名 |
| `e.oldValue` | 修改前的值 |
| `e.newValue` | 修改后的值 |
| `e.url` | 触发修改的页面 URL |
| `e.storageArea` | 被修改的存储对象 |

---

### 多标签页同步

**同步机制**：

```
标签页 A 修改数据
    ↓
更新 localStorage
    ↓
触发 storage 事件
    ↓
标签页 B、C、D... 收到事件
    ↓
读取新的 state
    ↓
更新本地 state
    ↓
重新渲染页面
```

**同步的状态**：

| 状态类型 | 是否同步 | 说明 |
|---------|---------|------|
| 用户数据 | 是 | users、matchmakers 等 |
| 业务数据 | 是 | matchRequests、chatMessages 等 |
| 系统设置 | 是 | settings、split ratios 等 |
| UI 状态 | 否 | 当前 Tab、滚动位置等 |
| 弹窗状态 | 否 | 打开的弹窗、表单输入 |
| 登录状态 | 是 | currentClient、currentMatchmaker |

**同步冲突处理**：

```
由于 localStorage 是最后写入为准：
1. 以 syncVersion 为准
2. 收到 storage 事件时比较版本号
3. 如果远程版本高 → 接受远程数据
4. 如果本地版本高 → 忽略事件，保持本地数据
5. 版本相同时不做处理
```

---

### 跨端同步机制

**综合预览端内部同步**：

```
三端（小程序/红娘/管理）共享同一个 window.state：

1. 任意端操作 → 更新 state
2. 调用 renderAll() 重新渲染所有视图
3. 各视图从最新 state 读取数据
4. 保证三端视图数据一致
```

**独立端口间同步**：

```
mini.html / matchmaker.html / admin.html 三个独立页面：

方式一：API 轮询（4秒）
- 每个页面每 4 秒拉取一次最新数据
- 比较版本号，有变化则更新

方式二：localStorage + storage 事件
- 数据变更写入 localStorage
- 其他标签页监听 storage 事件
- 实时性更高（几乎即时）

方式三：WebSocket（生产环境）
- 建立长连接
- 服务器主动推送变更
- 实时性最高
```

```
    其他用户/设备的浏览器
```

### 同步性能优化

**防抖与节流**：

```
频繁操作时的优化：
- 连续操作不每次都同步
- 使用防抖（debounce）延迟 300ms
- 防抖期间的多次操作合并为一次同步

示例：
用户快速连续发送 5 条消息
→ 防抖 300ms
→ 只同步一次（包含 5 条消息）
```

**增量同步**：

```
全量同步 vs 增量同步：

全量同步（当前实现）：
- 每次同步完整的 state JSON
- 实现简单
- 数据量大时较慢

增量同步（优化方向）：
- 只同步变更的部分
- 用 patch 格式描述变更
- 更快，流量更少
```

**本地存储压缩**：

```
localStorage 容量限制（通常 5MB）：
- state 过大时可能超限
- 可以考虑使用压缩算法（如 lz-string）
- 或只存储关键数据，非关键数据可重建
```
