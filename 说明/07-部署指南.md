# MatchMaker — 部署指南

本文档是当前项目真实部署流程。旧版文档中的手动递增版本号、手动 rsync、`app.js?v=1.0.x` 等流程已经废弃。当前标准流程是：

```text
本地改代码
本地检查
Git commit
Git push origin master
服务器执行 deploy/auto-deploy.sh
服务器 git pull
服务器生成自动版本号 dist/*.html
服务器重建单端口 Docker 网关
访问 `https://uk.sbbz.tech:21314/` 验证
```

## 5. 本地发布前检查

在本地项目目录执行：

```bash
cd /Users/x/code/matchmaker
git status --short
node --check app.js
node --check server/index.js
bash -n deploy/auto-deploy.sh
bash -n deploy/remote-deploy.sh
git diff --check
```

如果改了 uniapp：

```bash
cd /Users/x/code/matchmaker/uniapp
npm run build:h5
npm run build:mp-weixin
```

如果改了静态 HTML 模板，可以本地验证自动版本号生成：

```bash
cd /Users/x/code/matchmaker
npm run render:static
grep -R "app.js?v=" -n dist | head
```

`dist/` 是生成物，被 `.gitignore` 忽略，不提交。

## 7. 服务器部署

执行：

```bash
ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech \
  'cd /opt/matchmaker && bash deploy/auto-deploy.sh'
```

脚本当前流程：

```text
1. cd /opt/matchmaker
2. git pull origin master
3. 计算变更文件
4. 查找 node/nodejs
5. 执行 scripts/render-static.mjs，生成 dist/*.html 和 dist/version.json
6. 如果 uniapp/ 有变化，执行 npm install 和 npm run build:h5
7. 如果 server/ 有变化，docker compose build api 并重启 api
8. 重启 HTTP 和 HTTPS 前端 Nginx 容器
9. 写入 /var/log/matchmaker-deploy.log
```

注意：如果刚刚修改的是 `deploy/auto-deploy.sh` 自身，第一次执行时 shell 可能仍按旧脚本继续跑。此时再执行一次部署脚本即可。

## 8. 自动版本号

HTML 模板中写：

```html
<script src="/app.js?v=__ASSET_VERSION__"></script>
```

部署时运行：

```bash
node scripts/render-static.mjs
```

输出：

```text
dist/index.html
uniapp/dist/build/h5/
dist/matchmaker.html
dist/admin.html
dist/version.json
```

`__ASSET_VERSION__` 会替换成当前 Git 短提交号，例如：

```html
<script src="/app.js?v=<Git短提交号>"></script>
```

确认服务器生成结果：

```bash
ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech \
  'cd /opt/matchmaker && git rev-parse --short HEAD && grep -R "app.js?v=" -n dist | head'
```

两边版本号应一致。

## 11. 常见问题

### 页面仍是旧 JS

排查：

```bash
curl -sk https://uk.sbbz.tech:21314/ | grep 'app.js?v='
ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech \
  'cd /opt/matchmaker && git rev-parse --short HEAD && grep -R "app.js?v=" -n dist | head'
```

修复：

```bash
ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech \
  'cd /opt/matchmaker && bash deploy/auto-deploy.sh'
```

### 502 Bad Gateway

看 API：

```bash
ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech \
  'docker ps | grep matchmaker-api || true && docker logs --tail=100 matchmaker-api'
```

重建 API：

```bash
ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech \
  'cd /opt/matchmaker && docker compose -f compose.yml -f compose.ssl.yml up -d --build api'
```

重启 Nginx 前端容器：

```bash
ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech \
  'cd /opt/matchmaker && docker compose -f compose.yml -f compose.ssl.yml restart web web-mini web-matchmaker web-admin web-ssl web-mini-ssl web-matchmaker-ssl web-admin-ssl'
```

### API 重建后仍 502

常见原因是 Nginx 容器仍解析到旧 API 容器 IP。重启前端 Nginx 容器即可。

### 本机 curl HTTPS 异常

如果本机网络把 `uk.sbbz.tech` 解析到异常地址，可能出现 TLS EOF 或连接超时。此时以浏览器实际访问和服务器本机 `127.0.0.1` 检查为准。

### uniapp 构建有 Sass 警告

Sass deprecated warnings 通常不阻塞。只要 `npm run build:h5` 和 `npm run build:mp-weixin` 退出码为 0，就算构建通过。

## 12. 回滚

如果新提交上线后出现严重问题：

```bash
ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech
cd /opt/matchmaker
git log --oneline -10
git revert 有问题的提交
bash deploy/auto-deploy.sh
```

不要直接 `git reset --hard`，除非明确知道会丢弃哪些提交并已经确认。

## 13. 数据库备份

备份：

```bash
ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech \
  'cd /opt/matchmaker && mkdir -p backup/postgres && docker exec matchmaker-postgres pg_dump -U matchmaker -d matchmaker -Fc -f /backup/backup-$(date +%F_%H%M%S).dump'
```

查看备份：

```bash
ssh -o StrictHostKeyChecking=no root@uk.sbbz.tech \
  'ls -lh /opt/matchmaker/backup/postgres | tail'
```

恢复前必须先确认要恢复的文件名。恢复会覆盖数据库，谨慎执行。
