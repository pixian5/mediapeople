# 业务逻辑审计与防错清单

本文是当前项目的权威业务逻辑说明。若其它旧文档与本文冲突，以本文为准。

最后审计日期：2026-07-08

## 一、角色和端口

| 角色 | 线上入口 | 主要能力 |
|---|---|---|
| 会员 | `https://uk.sbbz.tech:9446/` | 注册登录、资料维护、绑定红娘、成为红娘 VIP、浏览异性、申请牵线、聊天 |
| 红娘 | `https://uk.sbbz.tech:9447/` | 处理牵线请求、打开私聊、打开三方群、开关会员互聊、审核资料 |
| 管理员 | `https://uk.sbbz.tech:9448/` | 机构、红娘、客户、分成、兑换码、成交数据 |
| 综合预览 | `https://uk.sbbz.tech:9445/` | 同页预览多角色界面 |

## 二、核心数据对象

| 对象 | 表/字段 | 业务含义 |
|---|---|---|
| 会员 | `users.raw` | 用户完整资料、VIP 红娘、委托红娘、按红娘隔离的资料版本 |
| 红娘 | `matchmakers.raw` | 红娘账号、机构、推荐码 |
| 牵线请求 | `match_requests.raw` | 一次“会员 A 想认识会员 B，由红娘 M 负责”的业务单 |
| 聊天线程 | `chat_threads.raw` | 一条独立会话，必须由 `type + requestId + participants` 唯一区分 |
| 聊天消息 | `chat_messages.raw` | 消息只属于一个 `threadId`，严禁跨线程混入 |
| 兑换码 | `promo_codes.raw` | VIP 兑换或推广归属 |
| 成交 | `deals.raw` | VIP 支付/模拟成交流水 |

## 三、会员和红娘绑定逻辑

会员可以同时绑定多个红娘。

| 字段 | 含义 |
|---|---|
| `vipMatchmakerIds` | 会员已经付费成为 VIP 的红娘列表 |
| `delegatedMatchmakerIds` | 会员资料委托发布/审核的红娘列表 |
| `referralMatchmakerId` | 默认或首个归属红娘，用于兼容旧逻辑 |
| `profileByMatchmaker` | 按红娘隔离的资料版本，每个红娘看到和审核自己的资料副本 |

规则：

- 会员点击某个异性会员申请牵线时，必须选择目标会员绑定的红娘之一。
- 如果申请人还不是该红娘的 VIP，先通过该红娘推荐码开通 VIP。
- 成为红娘 A 的 VIP，不等于自动成为红娘 B 的 VIP。
- 修改资料后，会给相关红娘生成 `pending` 草稿。
- “一键同步到所有红娘”只能同步到会员已绑定/已选择的红娘，不应覆盖无关红娘的数据。

## 四、牵线请求状态

当前请求字段：

| 字段 | 含义 |
|---|---|
| `status` | 展示用状态，来自联系标记推导 |
| `maleContacted` | 是否已标记联系男方 |
| `femaleContacted` | 是否已标记联系女方 |
| `memberChatEnabled` | 红娘是否允许双方会员直接互聊 |

状态推导：

| 条件 | `status` |
|---|---|
| 都未标记 | `待红娘联系` |
| 只标记男方 | `联系男方` |
| 只标记女方 | `联系女方` |
| 男女都标记 | `来和双方对话` |

重要边界：

- 现在工作台按钮“联系男方/联系女方”的主语义是打开对应一对一聊天，不是标记进度。
- `/api/matchmaker/requests/:id/contacted` 仍保留为进度标记接口，但不应被“打开私聊”按钮偷偷调用。
- 如果未来需要恢复进度标记，必须新增明确按钮，例如“标记已联系男方”，不能复用“联系男方”。

## 五、聊天线程类型

这是最重要的防错规则。

| `type` | 参与者 | 谁能发消息 | 显示位置 | 业务含义 |
|---|---|---|---|---|
| `member_matchmaker` | 1 个红娘 + 1 个会员 | 该红娘和该会员 | 会员“和红娘聊天”、红娘聊天列表 | 一对一私聊 |
| `matchmaker_group` | 1 个红娘 + 2 个会员 | 三个参与者 | 会员“三方群聊”、红娘聊天列表/请求按钮 | 红娘把双方拉进群 |
| `member_member` | 2 个会员 | 两个会员 | 会员“会员互聊” | 红娘开通后双方直接聊 |

硬约束：

- `member_matchmaker` 永远只能有 2 个参与者。
- `matchmaker_group` 永远必须有 3 个参与者：红娘、发起会员、目标会员。
- `member_member` 永远只能有 2 个会员参与者。
- 消息只能按 `threadId` 查询和写入，不能按 `requestId` 合并不同线程消息。
- 任何 UI 标题、按钮、接口返回都必须能让用户分辨“一对一”“三方群聊”“会员互聊”。

## 六、牵线创建时应生成的线程

会员申请牵线成功后，后端应确保存在：

| 线程 | 创建函数 | 用途 |
|---|---|---|
| 男方与红娘一对一 | `buildMemberMatchmakerThreads()` | 红娘联系男方 |
| 女方与红娘一对一 | `buildMemberMatchmakerThreads()` | 红娘联系女方 |
| 红娘与双方三方群 | `buildMatchmakerGroupThread()` | 红娘和双方共同沟通 |

`member_member` 不在申请时开放，它只在红娘开启 `memberChatEnabled` 后可用。

## 七、红娘端按钮语义

| 按钮 | 正确行为 | 禁止行为 |
|---|---|---|
| 联系男方 | 打开男方 `member_matchmaker` 一对一线程 | 不能打开女方、三方群、会员互聊；不能默认标记进度 |
| 联系女方 | 打开女方 `member_matchmaker` 一对一线程 | 不能打开男方、三方群、会员互聊；不能默认标记进度 |
| 来和双方对话 | 打开 `matchmaker_group` 三方群线程 | 不能复用任何 `member_matchmaker` 私聊 |
| 开通双方沟通 | 设置 `memberChatEnabled=true`，确保 `member_member` 存在 | 不能把红娘加入 `member_member` |
| 关闭双方沟通 | 设置 `memberChatEnabled=false` | 不能删除历史消息，可隐藏/禁止发送 |

## 八、会员端聊天展示

会员端应分三类展示：

| 区域 | 展示线程 |
|---|---|
| 和红娘聊天 | `member_matchmaker` |
| 三方群聊/会员互聊区域 | `matchmaker_group` 和 `member_member`，但标题必须区分 |
| 会员互聊 | 只有 `memberChatEnabled=true` 时展示 `member_member` |

uniapp 聊天列表接口 `/api/client/chat/threads` 必须对 `matchmaker_group` 返回组合标题，例如“李莉、周晴（三方群聊）”，不能只返回“李莉”，否则用户会误以为是私聊。

## 九、消息发送权限

后端统一接口：

```text
POST /api/chat/threads/:id/messages
```

发送前必须检查：

- 当前 token 角色和 ID 必须在 `thread.participants` 中。
- `thread.status` 必须是 `active`。
- 如果是 `member_member`，必须额外检查 `request.memberChatEnabled=true`。
- `matchmaker_group` 不受 `memberChatEnabled` 限制，因为它是红娘参与的三方沟通，不是会员直接互聊。

## 十、资料审核逻辑

资料按红娘隔离：

| 操作 | 结果 |
|---|---|
| 会员修改资料 | 更新用户基础字段，并给选定红娘写入 `profileByMatchmaker[matchmakerId].draft` |
| 红娘审核通过 | `draft` 变为 `published`，状态为 `approved` |
| 红娘退回 | 保留 `draft`，状态为 `rejected` |
| 浏览异性资料 | 优先展示目标会员在对应红娘下的 `published`，否则展示基础资料 |

防错规则：

- 红娘 A 审核通过，不应影响红娘 B 的发布资料。
- 资料更新不应直接覆盖所有红娘的 `published`。
- “一键同步”是批量创建/更新草稿，不是绕过审核直接发布。

## 十一、VIP 和支付逻辑

| 场景 | 规则 |
|---|---|
| 输入兑换码 | 可成为兑换码指定红娘的 VIP，或通用 VIP |
| 选择目标会员申请牵线 | 若不是目标红娘 VIP，先成为该红娘 VIP |
| 多红娘 VIP | `vipMatchmakerIds` 追加，不覆盖 |
| 成交流水 | 每次开通写入 `deals` |

防错规则：

- 会员已是红娘 A 的 VIP，不能绕过付费申请红娘 B 绑定的目标会员。
- 目标会员必须绑定所选红娘，否则申请牵线应失败。

## 十二、管理员逻辑

管理员可维护：

- 机构。
- 红娘。
- 分成比例，总和必须等于 100。
- 兑换码。
- 模拟成交。
- 客户列表。

管理员不应直接绕过聊天权限发送业务消息；管理员 token 目前可访问聊天接口是开发期能力，正式上线前应收紧。

## 十三、部署和缓存逻辑

部署流程：

```text
本地改代码 → git commit/push → 服务器 auto-deploy.sh 拉取 → render-static 生成 dist/*.html → 重启 Nginx 容器
```

关键点：

- HTML 中 `app.js?v=__ASSET_VERSION__` 部署时替换成 Git 短提交号。
- Nginx 容器以文件 bind mount 挂载 `app.js` 和 `styles.css`。
- Git 更新文件后，如果不重启容器，文件型 bind mount 可能继续指向旧 inode，导致“版本号新但脚本旧”。
- 因此 `auto-deploy.sh` 必须完整跑到“重启 Nginx 容器”步骤，不能在中途退出。

验收命令：

```bash
curl -k -sS https://uk.sbbz.tech:9447/ | rg "app.js\\?v="
curl -k -sS "https://uk.sbbz.tech:9447/app.js?v=$(git rev-parse --short HEAD)" | rg "matchmaker_group"
```

## 十四、每次改聊天逻辑必测清单

线上浏览器实测，不用本地网站：

1. 打开红娘端 `https://uk.sbbz.tech:9447/`。
2. 登录李莉。
3. 找到一个“男方申请认识女方”的请求。
4. 点“联系男方”，标题必须是“男方名（一对一沟通）”，线程类型必须是 `member_matchmaker`。
5. 点“联系女方”，标题必须是“女方名（一对一沟通）”，线程类型必须是 `member_matchmaker`。
6. 点“来和双方对话”，标题必须包含“三方群聊”，线程类型必须是 `matchmaker_group`。
7. 在三方群发一条唯一文本，线上 state 查询必须只在 `matchmaker_group` 找到该消息。
8. 再点“联系男方”，不应看到刚发的三方群消息。
9. 开通双方沟通，会员端应出现 `member_member`。
10. 关闭双方沟通，会员端不应允许继续发送 `member_member` 消息。

## 十五、代码审计发现和当前处理

已修复：

- 红娘“来和双方对话”曾错误复用 `member_matchmaker` 私聊，已改为独立 `matchmaker_group`。
- 服务器自动部署曾在重启 Nginx 前退出，导致线上脚本旧缓存，已修复 `deploy/auto-deploy.sh`。
- uniapp 聊天列表曾把三方群当作单个“otherUser”显示，已改为组合群聊标题。
- 本地/离线兜底创建牵线请求时曾只创建一对一线程，未创建三方群，已补齐 `matchmaker_group`。
- uniapp 详情页曾只判断“是否任意 VIP”，未按目标会员绑定红娘选择和开通对应红娘 VIP，已改为按所选红娘申请牵线。
- 会员注册时如果未传 `delegatedMatchmakerIds` 曾保存为空数组，导致该会员可见但不能被正常申请牵线，已改为默认绑定全部红娘。
- uniapp 资料编辑传 `avatar`，后端资料接口未保存头像，已支持 `photo/avatar` 写入。
- uniapp 请求封装曾忽略旧接口 `{ error }` 错误体，已兼容显示真实错误。
- `/api/state` 曾允许会员/红娘全量覆盖 state，已收紧为仅管理员可写；普通页面不再自动全量写回远程 state。

仍需后续产品确认：

- “联系男方/联系女方”现在是打开私聊，不再标记进度；如果业务仍需要进度标记，应新增明确按钮。
- 请求卡片里的“来和双方对话”目前按状态显示；三方群线程已存在，也可以从聊天列表进入。是否让该按钮对所有牵线请求始终显示，需要产品确认。
- 管理员开发期 token 可访问聊天发送接口，正式上线前应收紧权限。
- `GET /api/state` 当前仍是开发期公共状态接口，会返回脱敏后的全量演示数据；正式上线前应改为按角色拆分接口。
- 会员查看微信目前仍有“全局 VIP”兼容逻辑，正式产品应进一步按红娘 VIP 或牵线状态细分。

## 十七、跨模块一致性规则

### 1. 静态端和 uniapp 必须遵守同一业务规则

- 申请牵线必须带 `targetUserId` 和最终选定的 `matchmakerId`。
- 如果目标会员绑定多个红娘，前端必须让会员选择一个红娘。
- 如果申请人不是该红娘 VIP，必须先成为该红娘 VIP。
- 前端不能只用 `user.vip` 判断是否可申请，必须看 `vipMatchmakerIds` 是否包含所选红娘。

### 2. 后端必须提供前端完成业务所需的信息

- 会员详情接口必须返回目标会员绑定的红娘列表 `delegatedMatchmakers`。
- 聊天列表接口必须返回足够区分线程类型的标题。
- 资料接口必须接收前端实际发送的字段，例如 `avatar/photo`。
- 旧格式 `{ error }` 和新格式 `{ code, message }` 应在客户端都能正确提示。

### 3. 开发期全量 state 只能作为管理员工具

- 业务操作必须走细粒度 API。
- 会员和红娘不能调用全量 `PUT /api/state`。
- 普通页面卸载时不能把本地 state 自动写回服务器。
- 管理员重置数据必须带管理员 token。

### 4. 注册默认值必须让后续流程可继续

- 新会员如果没有显式选择红娘，默认绑定全部红娘，避免“可见但不能申请牵线”。
- 新会员默认不是任何红娘 VIP，申请时按目标红娘付费开通。
- 新红娘由后台添加时可作为演示账号无密码登录；正式上线需要求密码。

## 十八、全盘回归清单

每次大改至少抽验以下路径：

1. 会员注册后能登录，`delegatedMatchmakerIds` 不为空。
2. 会员资料保存后，姓名、年龄、城市、职业、微信、自我介绍、择偶要求、头像字段能在 `/api/client/me` 看到。
3. 首页异性列表能打开详情，详情页能看到目标会员绑定红娘。
4. 选择目标红娘申请牵线时，如果不是该红娘 VIP，会先开通该红娘 VIP。
5. 牵线请求创建后，后端存在两条 `member_matchmaker` 和一条 `matchmaker_group`。
6. 红娘工作台能看到该请求，能分别打开男方/女方私聊，能打开三方群。
7. 红娘开通会员互聊后，会员端出现 `member_member`，关闭后不能继续发送会员互聊消息。
8. 红娘资料审核只影响该红娘下的 `profileByMatchmaker[matchmakerId]`。
9. 管理员添加机构、添加红娘、修改分成、生成兑换码、模拟成交均需要管理员 token。
10. `auto-deploy.sh` 完整执行后，线上 HTML 的 `app.js?v=` 与 Git 短提交一致，Nginx 容器已重启。
