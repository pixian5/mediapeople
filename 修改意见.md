# matchmaker 项目 uni-app 全栈优化重构方案
基于项目仍处于开发测试阶段、演示数据可重构的现状，本方案采用**「前端全面重构 + 后端同步合理化改造」**的思路：借前端迁移 uni-app 的窗口，一次性解决现有技术债，构建可长期迭代的标准前后端分离架构；同时严格控制改动边界，不推翻核心业务逻辑与基础设施，投入产出比最优。

## 一、方案总则与变更边界
### 1.1 核心目标
1. 前端：一套代码输出 H5 + 微信小程序双端，替代原有原生 HTML 客户端，体验与可维护性全面升级
2. 后端：同步完成接口从「整包状态同步」到「细粒度 RESTful」的演进，根除全表 truncate、大包传输等技术债
3. 架构：形成标准的前后端分离体系，权限隔离清晰、接口职责单一，支撑后续红娘端、支付、通知等功能迭代
4. 部署：无缝兼容现有 Docker + Nginx 架构，自动化部署流程保持一致，运维成本不增加

### 1.2 合理变更范围（本次同步优化）
所有变更均为解决现有明确技术债、提升长期可维护性，不做过度设计：
1. **接口层**：废弃 `GET /api/state` / `PUT /api/state` 整包读写模式，按角色拆分为细粒度单资源接口
2. **数据层**：移除 `syncNormalizedState()` 全表 truncate 重写逻辑，改为业务接口直接增量操作对应物理表
3. **鉴权层**：自实现 HMAC Token 升级为标准 JWT，支持过期时间、刷新机制，生态兼容性更好
4. **管理员体系**：从 `.env` 单密码升级为独立 `admins` 数据表，支持多管理员、密码哈希存储
5. **前端体验**：接入微信小程序原生能力（一键登录、头像昵称授权、分享），优化注册转化

### 1.3 保留不变的基础（避免推倒重来）
1. 核心业务逻辑与产品流程：注册登录、资料浏览、牵线申请、红娘对接、VIP 兑换、后台管理全流程不变
2. 技术栈底座：后端 Node.js + Express、数据库 PostgreSQL、部署 Docker + Nginx、服务器架构全部保留
3. 核心业务表结构：`users` / `matchmakers` / `agencies` / `match_requests` / `deals` / `promo_codes` 主体字段不变
4. 部署基础设施：端口规划、SSL 证书、反向代理配置、自动化部署脚本整体框架不变

## 二、整体技术架构
采用标准三层前后端分离架构，各层职责清晰、解耦独立：
1. **表现层**：uni-app 前端工程，一套代码编译为 H5 + 微信小程序，负责页面渲染、交互、状态管理
2. **接口层**：Express 后端，按角色拆分 RESTful 接口，统一鉴权、参数校验、错误处理
3. **数据层**：PostgreSQL 数据库，业务表作为唯一主数据源，`app_state` JSONB 表降级为可选快照备份

整体数据流：前端按页面按需调用对应接口 → 后端鉴权校验后操作业务表 → 返回精准数据 → 前端局部更新页面状态。彻底替代「前端拉全量状态 → 本地修改 → 整包回传 → 后端全量覆盖」的低效模式。

## 三、前端 uni-app 详细设计
### 3.1 技术栈选型（固定标准）
| 分类 | 选型 | 选型说明 |
|------|------|----------|
| 基础框架 | uni-app（Vue3 + Vite + 组合式 API） | 官方主推稳定版，构建性能优，长期维护有保障 |
| UI 组件库 | uView Plus | 社交类组件丰富，表单、卡片、弹窗等开箱即用 |
| 状态管理 | Pinia | Vue3 官方推荐，轻量易用，仅管理全局共享状态 |
| 请求方案 | 基于 `uni.request` 二次封装 | 跨端统一，内置拦截器、错误处理、Token 自动挂载 |
| 样式方案 | SCSS + rpx 单位 + BEM 命名规范 | 多端自适应，样式可维护性强 |
| 工程规范 | ESLint + 目录结构强制约定 | 统一代码风格，降低后续维护成本 |

### 3.3 核心模块设计
#### 3.3.1 请求与鉴权层
1. 统一封装请求实例，内置请求拦截器与响应拦截器
2. 鉴权逻辑：自动从 Pinia + 本地存储读取 JWT Token，挂载 `Authorization: Bearer <token>` 请求头
3. 错误处理：401 自动清除登录态并跳转登录页；业务错误统一弹出提示；网络异常给出通用提示
4. 基础地址自动切换：开发环境走 Vite 代理；H5 生产环境用相对路径 `/api`；小程序生产环境用完整 HTTPS 域名

#### 3.3.2 状态管理与本地存储
1. Pinia 仅存储：用户基础信息、登录状态、会员身份、筛选条件缓存
2. 本地存储统一使用 `uni.setStorageSync` 封装，Key 命名保持规范，登录态持久化
3. 页面级数据全部在页面内管理，不放入全局 Store，避免状态冗余
4. 敏感数据（身份证、密码）永不存入本地存储

#### 3.3.3 样式与设计体系
1. 1:1 复用原有视觉规范，主题色、字号、间距、圆角保持一致，不做产品级改版
2. 统一使用 rpx 单位，基于 750rpx 设计稿标准，保证多端屏幕自适应
3. 抽离全局 SCSS 变量：主题色、文字色、间距、圆角，全局可引用
4. 页面样式默认 scoped 隔离，避免全局污染

### 3.4 核心页面功能设计
所有页面业务逻辑与字段与原项目保持一致，仅数据来源从整包 state 改为对应细粒度接口。

1. **登录注册页**
   - 支持账号密码登录/注册，校验规则与原项目一致
   - 小程序端新增「微信一键登录」「头像昵称快速填写」入口，优化注册转化
   - 登录成功后写入 Store 与本地存储，跳转首页

2. **首页（资料列表）**
   - 顶部筛选栏：性别、年龄、地区筛选，筛选条件本地缓存
   - 瀑布流/列表展示用户卡片，支持下拉刷新、上拉分页加载
   - 调用用户列表接口，按需加载，不拉全量数据
   - 未登录可浏览，点击核心操作引导登录

3. **资料详情页**
   - 完整展示用户全部资料字段
   - VIP 权限控制：非 VIP 隐藏联系方式，引导开通
   - 底部固定操作栏：申请牵线按钮，点击唤起申请弹窗/跳转申请页

4. **我的页面**
   - 顶部个人信息栏：头像、昵称、会员标识、基本资料
   - 功能菜单：编辑资料、实名认证、VIP 兑换、我的牵线、设置
   - 数据实时从后端拉取，保证最新状态

5. **资料编辑页**
   - 分模块展示可编辑字段，表单校验规则与原项目一致
   - 头像上传统一使用 `uni.chooseImage`，兼容双端
   - 提交后调用个人资料更新接口，成功后同步更新全局状态

6. **VIP 兑换页**
   - 兑换码输入 + 权益说明
   - 调用兑换接口，成功后更新用户会员状态，给出成功提示

7. **牵线申请与记录页**
   - 申请页：带入对方信息，填写备注后提交
   - 记录页：展示本人提交的所有牵线请求，按时间倒序，展示处理状态

### 3.5 小程序原生能力接入
借重构机会一次性接入，提升小程序端体验：
1. 微信一键登录：通过 `wx.login` 换取 openid，支持快速注册/登录
2. 头像昵称填写：使用微信官方头像选择、昵称输入组件，降低注册门槛
3. 分享能力：支持分享资料卡片给好友、分享到朋友圈
4. 图片预览：头像、相册支持全屏预览手势
5. 后续可扩展：微信支付、订阅消息通知、客服消息

## 四、后端接口重构设计
### 4.1 接口设计原则
1. **RESTful 规范**：资源导向，语义清晰，GET 查询、POST 创建、PATCH 更新、DELETE 删除
2. **权限隔离**：客户端、红娘端、管理端接口分路径隔离，各自鉴权，避免越权
3. **细粒度单资源**：每个接口只负责一类数据，按需返回，杜绝大包冗余传输
4. **统一返回格式**：所有接口统一 `{ code, message, data }` 结构，前端统一处理
5. **参数校验**：所有写接口做后端参数校验，拦截非法输入

### 4.2 接口清单（按角色拆分）
#### 4.2.1 公共认证接口
- `POST /api/auth/client/login`：客户端账号密码登录
- `POST /api/auth/client/register`：客户端账号密码注册
- `POST /api/auth/matchmaker/login`：红娘登录
- `POST /api/auth/admin/login`：管理员登录
- `POST /api/auth/logout`：统一登出

#### 4.2.2 客户端接口（前缀 `/api/client`）
- `GET /api/client/me`：获取当前登录用户个人资料
- `PATCH /api/client/me`：更新个人资料
- `POST /api/client/real-name`：提交实名认证
- `GET /api/client/profiles`：分页获取异性资料列表，支持筛选条件
- `GET /api/client/profiles/:id`：获取指定用户详情
- `POST /api/client/match-requests`：提交牵线申请
- `GET /api/client/match-requests`：获取我的牵线申请列表
- `POST /api/client/vip/redeem`：使用兑换码开通 VIP

#### 4.2.3 红娘端接口（前缀 `/api/matchmaker`，二期扩展）
- `GET /api/matchmaker/workbench`：工作台数据概览
- `GET /api/matchmaker/requests`：获取分配给我的牵线请求列表
- `GET /api/matchmaker/requests/:id`：获取牵线请求详情（含双方联系方式）
- `PATCH /api/matchmaker/requests/:id/contacted`：标记已联系

#### 4.2.4 管理端接口（前缀 `/api/admin`，二期扩展）
- `GET /api/admin/metrics`：数据概览统计
- `GET /api/admin/users`：用户列表，支持分页、搜索
- `PATCH /api/admin/users/:id`：编辑/禁用用户
- `GET /api/admin/matchmakers`：红娘列表
- `POST /api/admin/matchmakers`：新增红娘账号
- `GET /api/admin/agencies`：机构列表
- `POST /api/admin/agencies`：新增机构
- `GET /api/admin/promo-codes`：兑换码列表
- `POST /api/admin/promo-codes`：生成兑换码
- `GET /api/admin/deals`：成交记录
- `GET /api/admin/splits`：分成配置
- `PATCH /api/admin/splits`：更新分成配置

### 4.3 鉴权体系升级
1. 从自实现 HMAC Token 升级为标准 JWT（JSON Web Token）
2. Token 包含字段：用户ID、角色、签发时间、过期时间
3. 支持过期自动失效，后续可扩展刷新 Token 机制
4. 不同角色 Token 隔离，客户端 Token 无法调用红娘/管理端接口，后端中间件统一校验角色权限
5. 密钥通过 `.env` 配置，不硬编码到代码中

### 4.4 兼容过渡方案
为降低上线风险，保留旧接口作为兜底：
1. `GET /api/state` 保留，但标记为废弃，仅用于回滚与兼容，不再作为前端主数据源
2. `PUT /api/state` 保留，前端全面切换后下线
3. 切换期内新旧接口并行，数据都写入同一套业务表，保证数据一致性

## 五、数据库层优化设计
### 5.1 核心调整
1. **主数据源切换**：业务表（`users`、`match_requests` 等）作为唯一真实数据源，所有读写直接操作业务表
2. **移除全量重写逻辑**：彻底删除 `syncNormalizedState()` 中 `truncate table ... cascade` + 全量插入的逻辑
3. **JSONB 表降级**：`app_state` 表从主数据降级为可选快照表，用于备份、调试，不再参与业务读写
4. **新增管理员表**：新增 `admins` 表，存储管理员账号、密码哈希、创建时间，替代 `.env` 单密码模式

### 5.2 数据一致性保障
1. 所有写操作通过数据库事务保证原子性
2. 关键字段添加唯一约束（如用户账号、兑换码），避免重复数据
3. 敏感字段（密码哈希、身份证号）单独存储，查询接口默认脱敏返回

### 5.3 可选后续扩展（本期不强制）
- 新增 `audit_logs` 操作日志表，记录关键操作
- 新增登录日志、接口请求限流表

## 六、部署与运维适配
### 6.1 整体部署架构不变
1. 前端 H5 产物仍由 Nginx 容器托管，统一通过 21314 端口与反向代理配置提供服务
2. 后端 API 容器、PostgreSQL 容器架构不变，端口、数据目录保持原样
3. SSL 证书路径、挂载方式不变

### 6.2 H5 端部署适配
1. 执行 `npm run build:h5` 生成静态产物到 `dist/build/h5`
2. 产物直接替换原 Nginx 容器挂载的静态资源目录
3. 构建自动添加资源哈希，解决缓存更新问题，移除原手动版本号方案
4. 原有 gzip、缓存策略、`/api` 反代配置全部复用，无需修改

### 6.3 自动化部署脚本优化
在原有 `deploy.sh` 基础上新增前端构建步骤，整体流程保持一致：
1. 语法检查：后端 JS 语法校验 + 前端构建校验
2. 前端构建：安装依赖、执行 H5 构建
3. 代码同步：前端静态资源 + 后端代码同步到服务器
4. 服务重启：仅重启 API 容器，前端静态资源替换即生效
5. 健康检查：遍历所有端口校验服务正常

### 6.4 环境配置调整
`.env` 文件新增/调整字段：
- 保留数据库、JWT 密钥等核心配置
- 移除 `ADMIN_PASSWORD` 字段，管理员账号迁移至数据库
- 新增 Token 过期时间配置项

## 七、分阶段落地计划
### 阶段一：基础底座并行搭建（2-3天）
**前端**：初始化 uni-app 工程，配置组件库、请求封装、状态管理、全局样式、路由配置
**后端**：JWT 鉴权中间件、统一返回格式、参数校验框架、管理员表创建、接口基础框架搭建
**验收**：前后端登录接口联调通过，鉴权链路跑通

### 阶段二：客户端核心业务开发（4-5天）
**前端**：完成首页列表、详情页、我的页面、资料编辑、VIP兑换、牵线申请全部页面
**后端**：完成客户端全部细粒度接口开发，直连业务表，移除对 JSONB 主数据依赖
**验收**：客户端全流程（注册→登录→浏览→编辑→牵线→兑换）联调跑通，数据正确写入业务表

### 阶段三：多端适配与全量测试（2-3天）
1. 微信小程序端适配，修复平台差异，接入微信登录、分享等原生能力
2. 双端功能、样式、交互一致性测试
3. 接口权限、参数校验、异常场景测试
4. 部署脚本集成，线上环境灰度验证
5. 旧接口兼容测试，确认回滚路径可用

### 阶段四：平滑上线与旧逻辑下线（1天）
1. H5 端全量切换为新前端
2. 小程序端提交审核发布
3. 观察运行状态，保留旧接口兜底
4. 稳定运行 1-2 周后，正式下线 `PUT /api/state` 整包写入逻辑

### 阶段五：后续扩展（可选）
- 红娘端 uni-app 迁移
- 管理后台优化重构
- 微信支付、订阅消息接入
- 操作日志、限流等安全能力补充

## 八、风险控制与回滚方案
1. **数据风险**：所有表结构变更不删字段、只新增，保留兼容；上线前全量备份数据库
2. **接口风险**：新旧接口并行，前端可一键切换回旧版，出现问题分钟级回滚
3. **部署风险**：前端静态资源保留历史版本目录，可快速回退到上一版本
4. **小程序风险**：H5 端先上线验证，小程序端延后发布，保证核心业务可用

## 九、验收标准
### 前端验收
1. H5 + 微信小程序双端均可正常运行，核心流程无阻断
2. 所有页面视觉风格与原项目一致，字段完整无缺失
3. 登录态跨页面同步正确，权限控制生效
4. 长列表滑动流畅，无明显卡顿与白屏
5. 小程序原生能力（登录、分享）正常可用

### 后端验收
1. 所有细粒度接口功能正常，参数校验、权限校验生效
2. 数据直接写入业务表，无全表 truncate 操作
3. 鉴权体系正常，不同角色无法越权访问
4. 旧接口仍可正常使用，数据与新接口一致
5. 接口响应速度较原大包接口有显著提升

### 部署验收
1. H5 产物可无缝部署到原有服务器架构，替换后即可访问
2. 自动化部署脚本可一键完成全流程发布
3. 健康检查全部通过，所有端口服务正常
4. SSL 证书生效，HTTPS 访问正常
